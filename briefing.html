§1

Situation — what this codebase is

This repository is the starter for a three-day course exercise. The application does not exist yet; what exists is the brief, the raw material, and one architectural decision already taken.

The end state, per the README: an agent that watches three live disaster feeds (GDACS, USGS, ReliefWeb), filters out noise and assesses what remains — what happened, where, how bad, who is affected — and publishes a morning situation report to dashboard.html at 08:30 Singapore time. It runs on a schedule, unattended, and stays quiet when nothing has changed. How it does any of that is deliberately unspecified — that is the course.

One decision is already encoded, in the disabled workflow .github/workflows/sitrep.yml.disabled: a deterministic script decides whether anything changed, and a headless model call runs only if it did. In the file's own words — the model never decides whether to wake up. Everything built this week has to fit that shape.

§2

Course of action — what I would build first

First slice: GDACS only, end to end — fetch, filter to Orange/Red, render the dashboard, detect change. Thin, but it exercises every joint the final system needs.

Why GDACS and not the others: it is multi-hazard (the mission is HADR, not seismology), needs no approval or keys, and every event arrives with an alert level — a severity filter we get for free instead of inventing one. USGS is an earthquakes-only firehose that mostly needs discarding, and ReliefWeb's API is blocked on an appname approval that may not arrive this week. Both join later as their own slices.

  1. Fetch & normalise

    scripts/fetch-gdacs pulls the event list (following redirects — the one lesson already in docs/solutions/), and writes a small, stable, normalised state/events.json. Raw feed in, boring JSON out, no model anywhere.

  2. Filter to signal

    Keep Orange and Red alerts only — a defensible default until RFI–03 below is answered. The filter lives in the script as data, not in a prompt.

  3. Render the dashboard

    Generate dashboard.html deterministically from the normalised state: date, event count, one row per event — what, where, alert level, source link. Ugly is fine; honest is required.

  4. Detect change

    Diff the new normalised state against the last committed one and exit accordingly. This exact exit code is what sitrep.yml's TODO 1 is waiting for — the deterministic gate in front of any model call.

  5. Prove idempotence

    Definition of done: run the pipeline twice against an unchanged feed and get a byte-identical dashboard.html plus a "no change" exit. Checkable by a reviewer in under two minutes, as the slice template demands.

Deliberately out of scope for slice 1 USGS and ReliefWeb ingestion · cross-feed deduplication · model-written prose in the report · the cron schedule itself. Each is its own slice, sequenced after the walking skeleton works. Recorded here so nobody — human or agent — helpfully builds them anyway.
§3

Requests for information — ten questions before starting

Colour codes follow the GDACS convention: red blocks the first slice, orange blocks a later slice, green shapes quality but blocks nothing.

  1. Who reads the 08:30 report, and what decision does it drive?

    A duty officer deciding whether to escalate reads differently from a team wanting ambient awareness. Tone, length, and what counts as "assessed" all hang on this.

    redBlocks the PRD and every wording choice in the dashboard.

  2. Is coverage global, or weighted to a region?

    The feeds are global; the reader probably is not. A Singapore-anchored view (Asia-Pacific first, rest of world below the fold) changes the filter and the layout.

    redBlocks the noise filter — "signal" is undefined without a scope.

  3. What is the reportable-severity threshold, and who owns it?

    Slice 1 assumes GDACS Orange+. Is that right? What is the equivalent line for USGS (magnitude? the sig score? tsunami flag?) — and is the threshold config a human edits, or judgment the model applies?

    redBlocks the filter; the Orange+ default stands until answered.

  4. What exactly does "stays quiet when nothing has changed" mean?

    An untouched dashboard.html? A fresh page that says "no change as of 08:30"? Is there ever a push notification to a human, or is publishing the page the whole output?

    redBlocks the change-detection contract in slice 1, step 4.

  5. How much event identity do we owe in three days?

    The same earthquake arrives from GDACS and USGS under different IDs, and both feeds revise or delete events after publication. Is single-feed-at-a-time acceptable for the demo, or is cross-feed dedup part of the definition of done?

    orangeBlocks the multi-feed slice and the data model's key choice.

  6. Has a ReliefWeb appname been requested yet?

    Approval arrives by email on someone else's clock. If it will not land this week, is the no-approval RSS feed an acceptable stand-in for the demo — knowing it lacks the API's structured fields?

    orangeBlocks the ReliefWeb slice; request should go out today either way.

  7. Where does state live, and does history matter?

    The obvious three-day answer is committed JSON — git as the database, every morning's state reviewable in the log. Is that acceptable, and do past sitreps need to remain retrievable, or does today's page overwrite yesterday's?

    orangeBlocks change detection design and the repo's growth rate.

  8. Confirm the runtime: GitHub Actions, and which secrets?

    The disabled workflow implies Actions is the venue. Is an Anthropic API key available as a repo secret for the headless claude -p step, is there a spend ceiling — and what should 08:30 publish on a morning a feed is down or the run fails?

    orangeBlocks the scheduling slice and the failure-morning behaviour.

  9. How deep should "who is affected" go?

    The feeds carry population-impact hints (GDACS alert scores, per-event detail URLs). May the assessment step fetch those detail endpoints, or is it the model reasoning over the summary state only? Where is the line between deterministic rendering and model judgment in the report body?

    greenShapes assessment quality; slice 1 ships without it.

  10. What does Wednesday's demo actually have to show?

    A live run on stage, or the artefact trail of unattended overnight runs — commits at 08:30, a dashboard that updated itself, quiet days staying quiet? Which artefacts are assessed, and what separates "done" from "impressive"?

    greenShapes sequencing of days 2–3; nothing in slice 1 waits on it.
